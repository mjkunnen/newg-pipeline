# Plan 02-01: Central Config + Postgres Dedup Migration

## Objective

Replace hardcoded account lists and file-based dedup with a central config system and Postgres-backed dedup. This makes TikTok and Pinterest scrapers use the content API for seen-content tracking (same pattern as PPSpy), and moves all competitor/threshold configuration out of scraper source code.

## Context

- `@decarba-remixer/src/scraper/tiktok.ts` — hardcoded TIKTOK_ACCOUNTS, reads/writes scout/processed_tiktok.json for dedup
- `@decarba-remixer/src/scraper/pinterest.ts` — hardcoded BOARD_URL and SHEET_ID, reads Google Sheet for dedup
- `@decarba-remixer/src/scraper/ppspy.ts` — hardcoded "decarba" search term in URL
- `@decarba-remixer/src/index.ts` — writeToContentAPI() pattern already established (lines 47-100)
- `@decarba-remixer/config/settings.yaml` — existing config file (pipeline settings only, no scraper configs)
- Decision D-01: Postgres ON CONFLICT DO NOTHING handles idempotent inserts — no Redis or file dedup needed
- Decision D-02: Retire scout/processed_tiktok.json — TikTok scraper POSTs to content API instead
- Decision D-03: Pinterest dedup moves to Postgres, stop reading Google Sheet for seen-content
- Decision D-13: Create one config file per source in decarba-remixer/config/
- Decision D-14: PPSpy search term moves out of hardcoded URL into config

## Tasks

### Task 1: Create central source config files
**Type:** auto
**Done when:** Four JSON config files exist in `decarba-remixer/config/` — one per source (tiktok, pinterest, meta, ppspy) — each with competitor accounts/URLs, viral thresholds, and enable toggle.

Create:
- `decarba-remixer/config/tiktok-accounts.json` — competitor TikTok accounts list, MIN_REACH, MAX_AGE_DAYS, MAX_CAROUSELS
- `decarba-remixer/config/pinterest-boards.json` — board URL(s), MAX_NEW_PINS
- `decarba-remixer/config/meta-competitors.json` — competitor brand names/pages for Meta Ad Library search (placeholder list based on NEWGARMENTS market segment: Gen Z streetwear NL)
- `decarba-remixer/config/ppspy-settings.json` — search term(s), competitor brand names

All thresholds match current hardcoded values (do not change behavior, just externalize).

### Task 2: Refactor tiktok.ts to read config + write to content API
**Type:** auto
**Done when:** `tiktok.ts` reads account list and thresholds from `config/tiktok-accounts.json`, calls `writeToContentAPI()` after scraping (source="tiktok"), and no longer reads/writes `scout/processed_tiktok.json`. Dedup is purely Postgres ON CONFLICT. The `getProcessedIds()` and `saveProcessedId()` functions are removed.

Implementation notes:
- Import and call `writeToContentAPI()` from `../index.js` — but this creates a circular dep. Instead, inline the same fetch pattern directly in tiktok.ts (copy the pattern, use source="tiktok"). Or extract to a shared `lib/contentApi.ts`.
- Extract to `decarba-remixer/src/lib/contentApi.ts` — reusable module that both index.ts and individual scrapers can import.
- Remove PROCESSED_FILE constant and all file-based dedup functions.
- Keep the viral filter logic (MIN_REACH, MAX_AGE_DAYS) — just read thresholds from config instead of constants.

### Task 3: Refactor pinterest.ts to read config + write to content API
**Type:** auto
**Done when:** `pinterest.ts` reads board URL from `config/pinterest-boards.json`, calls shared `writeToContentAPI()` (source="pinterest"), and no longer reads the Google Sheet for dedup. The `getProcessedPinIds()` function and SHEET_CSV_URL constant are removed. MAX_NEW_PINS reads from config.

Note: Pinterest still doesn't have reach/engagement data from this source (it's a curated board, not algorithmic). The reach field stays 0. That's acceptable for v1.

### Task 4: Refactor ppspy.ts search term to read from config
**Type:** auto
**Done when:** `ppspy.ts` reads the search term (currently "decarba") from `config/ppspy-settings.json` instead of the hardcoded URL. The PPSPY_ADS_URL is constructed dynamically using the config value. Multiple search terms are supported (loop if array).

### Task 5: Update index.ts to use shared contentApi module
**Type:** auto
**Done when:** The `writeToContentAPI()` function in `index.ts` is replaced with an import from `src/lib/contentApi.ts`. No behavior change — just consolidation to single source of truth.

### Task 6: Remove processed_tiktok.json from git tracking
**Type:** auto
**Done when:** `scout/processed_tiktok.json` is added to `.gitignore` and removed from git tracking (`git rm --cached`). The daily-scrape.yml workflow no longer stages this file in its commit step.
